# Issue 6275: Runners summary: task_id should be Task ID

## Problem
In the Runner node execution details ("Runners summary"), the broker task
identifier is shown with the raw key `task_id` as its label, while every
sibling field uses a human-readable label ("Status", "Exit code",
"Execution mode", "Timeout (seconds)"). The screenshot in the issue shows
`task_id` rendered verbatim to the user.

## Root cause
`getExecutionDetails` in the runner mapper builds a
`Record<string, string>` where the keys are used directly as display
labels. The task id entry used the raw field name:

    details["task_id"] = taskID;   // web_src/src/pages/app/mappers/runner.tsx

## Fix
Use the human-readable label, consistent with the surrounding fields:

    details["Task ID"] = taskID;

Updated the matching expectation in
`web_src/src/pages/app/mappers/runner.spec.ts`.

## Scope / verification
- Searched the frontend for other `task_id` labels; the runner mapper is
  the only place the raw key was surfaced to users.
- The broker metadata key (`runner_broker_task_id`) and payload field
  (`payload.task_id`) are internal data keys, not display labels, so they
  are intentionally left unchanged.
- Full `vitest` run could not execute in this environment because the
  generated `src/api-client` (built from `api/swagger/*.json` via Go
  tooling) is not present. The change is a one-line label update matching
  the existing pattern and the spec was updated accordingly.

## Tradeoffs
- Minimal, targeted change: keeps the label consistent with peers and low
  risk. No refactor of the details map into a typed label mapping was done,
  since only this single key was affected; a broader label abstraction can
  be considered later if more raw keys appear.
