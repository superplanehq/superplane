# Plan: Run cancellation (#6196)

## Problem

Clicking **Stop** on a run in the UI (`useStopRunMutation` in
`web_src/src/ui/Runs/useRunInspectorActions.ts`) fans out a burst of client-driven
API calls:

1. `DescribeRun` to discover work (`fetchRunWorkForStop`)
2. `Promise.all` → one `CancelExecution` per running execution
3. one `DeleteNodeQueueItem` per queued item

A run with 3 active branches + 5 queued steps = **1 + 3 + 5 = 9 API calls**, and
there is a **race**: new executions/queue items can be created between the
`DescribeRun` snapshot and the cancels, so they survive the "stop".

There is no backend concept of "cancel this whole run". We need one — it is also a
hard prerequisite for [#6136](https://github.com/superplanehq/superplane/issues/6136)
(Run App component), which must cancel child runs when the parent run is cancelled.

## Current architecture (relevant parts)

- **Run** = `CanvasRun` / table `workflow_runs` (`pkg/models/canvas_run.go`).
  States: `started`, `finished`. Results: `passed`, `failed`, `cancelled`.
  A run has **no cancel action today**; it only becomes `finished` via the
  `RunFinalizer` worker, which derives the result from child execution results.
- **Executions**, **queue items**, and **events** each carry a `run_id` FK to
  `workflow_runs`.
- **RunFinalizer** (`pkg/workers/run_finalizer.go`) locks the run
  (`LockCanvasRunInTransaction`, `SELECT … FOR UPDATE`), checks
  `FindOpenCanvasRunWork` (active executions / queue items / pending events); when
  there is none it sets `state=finished`, computes `result`, publishes a run message.
  The `FOR UPDATE` lock deliberately blocks concurrent FK inserts of new work while
  finalizing.
- **Work producers**: `EventRouter` (events → queue items) and `NodeQueueWorker`
  (queue items → executions) create new work — these are where the race lives.
- **Existing precedent**: `cancelActiveExecutionsForDeletedNode`
  (`pkg/models/canvas_node.go:536`) already does node-scoped atomic cancellation in
  one transaction: bulk-cancel active executions (`cancelNodeExecutions`), bulk-delete
  queue items (`deleteQueueItemsForNode`), complete pending requests. `CancelRun` is
  the **run-scoped** version of exactly this.
- `CancelExecution` (`pkg/grpc/actions/canvases/cancel_execution.go`) additionally
  invokes each component's `action.Cancel(ctx)` hook for best-effort external cleanup
  (e.g. cancelling a remote CI job) and records `cancelled_by`.

## Proposed solution

Add a single, atomic **`CancelRun`** endpoint that replaces the client-side fan-out.

`POST /api/v1/canvases/{canvas_id}/runs/{run_id}/cancel`

### Flow

```mermaid
sequenceDiagram
    participant UI
    participant API as CancelRun handler
    participant DB as Postgres (workflow_runs)
    participant Comp as Component actions
    participant MQ as RabbitMQ

    UI->>API: POST /runs/{run_id}/cancel
    API->>DB: BEGIN; LockCanvasRun (FOR UPDATE)
    alt run already finished
        API-->>UI: 200 (idempotent no-op)
    else run started
        API->>DB: set cancelled_at, cancelled_by
        API->>DB: delete all queue items for run
        API->>DB: complete pending requests for run
        API->>DB: snapshot active execution IDs
        API->>DB: finalize run (state=finished, result=cancelled)
        API->>DB: COMMIT
        loop each active execution (best-effort, outside lock)
            API->>Comp: action.Cancel(ctx)
            API->>DB: execution -> finished/cancelled
        end
        API->>MQ: publish run + execution messages
        API-->>UI: 200
    end
```

### Backend changes

1. **Proto** (`protos/canvases.proto`): add `CancelRun` rpc near `DescribeRun`
   (`POST …/runs/{run_id}/cancel`) + `CancelRunRequest{canvas_id, run_id}` /
   `CancelRunResponse{}`. Optionally expose `cancelled_at` / `cancelled_by` on
   `CanvasRun`. Regenerate with `make pb.gen`.
2. **Migration** (`make db.migration.create NAME=add-cancelled-fields-to-workflow-runs`):
   add `cancelled_at timestamp` and `cancelled_by uuid` to `workflow_runs`. This is
   the cancellation marker + audit trail; the terminal state stays `finished` with
   `result=cancelled` (no new proto enum needed).
3. **Model** (`pkg/models/canvas_run.go`): add a run-scoped
   `CancelRunInTransaction(tx, workflowID, runID, cancelledBy)` mirroring
   `cancelActiveExecutionsForDeletedNode` but keyed on `run_id`:
   - `LockCanvasRunInTransaction` (FOR UPDATE); no-op if already `finished`.
   - set `cancelled_at`/`cancelled_by`.
   - run-scoped `cancelNodeExecutions` (add a `WHERE run_id = ?` variant) → returns
     the executions that need external `action.Cancel`.
   - run-scoped `deleteQueueItemsForNode` (`WHERE run_id = ?` variant).
   - complete pending requests for the run.
   - finalize inline: `state=finished`, `result=cancelled`, `finished_at=now`.
   - Make finalization honor the marker: a run with `cancelled_at != nil` finalizes
     to `result=cancelled` regardless of individual execution results (adjust
     `CalculateCanvasRunResultInTransaction` / `maybeFinalizeRun`).
4. **gRPC action** (`pkg/grpc/actions/canvases/cancel_run.go`): resolve acting user,
   run the model transaction, then **outside the lock** invoke `action.Cancel(ctx)`
   per snapshotted active execution (reuse the context-building logic from
   `cancel_execution.go`; extract a shared helper). Publish run + execution + memory
   messages after commit. Wire into `pkg/grpc/canvas_service.go`.
5. **Race guard on producers** (the part that makes this *correct*, not just fewer
   calls): the `FOR UPDATE` lock blocks inserts only while the txn is open — a message
   already in flight could still create work on a just-finished run. So make the two
   producers skip work for a run that is finished/cancelled:
   - `NodeQueueWorker` (`processNode`): before creating an execution from a queue item,
     bail if the run is finished/cancelled.
   - `EventRouter` (`processExecutionEvent`): before creating downstream queue items,
     bail if the run is finished/cancelled.
6. **Authorization** (`pkg/authorization/gateway_auth_rules.go`): add
   `POST …/runs/{run_id}/cancel` → `Resource: canvases`, `Action: update`,
   `DomainType: DomainTypeOrganization`, `ResourcePathParams: {CanvasIDPathParam}`
   (mirrors the `CancelExecution` rule).

### Frontend changes

- Regenerate the API client (`make pb.gen`), then collapse `useStopMutation` in
  `useRunInspectorActions.ts` to a **single** `canvasesCancelRun` call. Delete
  `fetchRunWorkForStop`, the `Promise.all` fan-out, and the now-unused
  `runningExecutionIds` / `queuedItems` stop plumbing. Keep `stopDisabled` derived from
  whether the run is still `STARTED`.
- Run `make format.js` and `make check.build.ui`.

### Forward-compat for #6136 (parent → child cascade)

There is **no parent/child run link today** (`parent_execution_id` was removed in
`20260614140634_remove-blueprints`). `CancelRun` is built as the reusable primitive so
#6136 only needs to:
- store a parent reference on the child run when the Run App component starts it, and
- have `CancelRun` publish a "run cancelled" message that a worker consumes to call
  `CancelRun` on child runs (cross-canvas, async — avoids nesting external calls inside
  the parent's transaction).

Keeping the core in a `CancelRunInTransaction` model function + a message on cancel
makes that a small, additive change rather than a rewrite.

## Pros / cons & tradeoffs

**Pros**
- One atomic API call replaces `1 + N + M`; no client-side snapshot race.
- Reuses proven patterns (`cancelActiveExecutionsForDeletedNode`, `LockCanvasRun`,
  `RunFinalizer`) — low novelty, low risk.
- Deterministic `result=cancelled` (not accidentally `passed`/`failed`).
- Clean seam for the #6136 cascade.

**Cons / tradeoffs**
- **External `action.Cancel` vs. lock hold.** Running N external HTTP cancels inside
  the `FOR UPDATE` transaction would hold the run lock for the slowest remote call.
  *Chosen tradeoff*: do all DB state changes (marker, delete, finalize) inside the
  lock and quickly; run best-effort `action.Cancel` **after** commit. Downside: a
  crash between commit and the external calls leaves remote work running, but the run
  is already correctly `cancelled` and finalized (same best-effort guarantee
  `CancelExecution` gives today).
- **Producer guards add a run lookup** on the hot path of `NodeQueueWorker` /
  `EventRouter`. Cheap (indexed PK read) and necessary to fully close the race.
- **New nullable columns** on `workflow_runs`; migration is additive and safe
  (no down-migration per repo convention).
- Marker (`cancelled_at`) is arguably redundant with `result=cancelled`, but it
  cleanly disambiguates "cancelling in progress", drives the producer guard, and gives
  an audit trail (`cancelled_by`).

## Testing

- Model: `CancelRunInTransaction` cancels active executions, deletes queue items,
  finalizes to `cancelled`; idempotent on an already-finished run; leaves other runs
  untouched.
- Action: `cancel_run_test.go` — authz, not-found, acting-user recorded, messages
  published, `action.Cancel` invoked per component execution.
- Race: a queue-item-created / execution-output event arriving after cancel does **not**
  create new work (producer-guard tests).
- E2E: stop a running multi-branch run → run ends `cancelled`, no orphan executions or
  queue items.
