# Issue #6219 — Duplicate API key name returns HTTP 500 instead of a validation/conflict error

## Problem

`POST /api/v1/api-keys` with a name that already exists in the organization
returns **HTTP 500** with a generic `"failed to create API key"` message.

The DB correctly rejects the duplicate via the unique index
`unique_api_key_in_organization` (SQLSTATE `23505`), and no second key is
created — so data stays consistent. But a foreseeable user input (reusing a
name, including one that only collides after whitespace trimming) surfaces as
an opaque server error.

**Root cause** — in `pkg/grpc/actions/apikeys/create_api_key.go:106-108`, the
transaction error (which carries the unique-constraint violation) is blanket
wrapped as `grpcerrors.Internal(...)`. There is no special-casing for the
`23505` violation.

```
db.Transaction(...)  ──► pgconn.PgError{23505, unique_api_key_in_organization}
                          │
                          └──► grpcerrors.Internal(err, "failed to create API key")  ──► HTTP 500 ❌
```

## Expected

A duplicate name should return a client-visible **conflict** response
(`AlreadyExists` → HTTP 409) explaining the name is already in use.

## Approach

Follow the pattern already established in this codebase for the same class of
problem (`Canvas` and `CanvasFolder` duplicate names):

1. A sentinel error in `pkg/models`.
2. A mapper that converts a `pgconn.PgError` with the matching
   `ConstraintName` into that sentinel.
3. The gRPC action inspects the sentinel with `errors.Is` and returns
   `grpcerrors.AlreadyExists`.

This keeps the DB as the single source of truth for uniqueness (no
pre-check / TOCTOU race) and reuses the sanitizer path that already maps
`codes.AlreadyExists` → HTTP 409.

### Reference patterns (existing, unchanged)

- `pkg/models/canvas.go` — `MapCanvasNameUniqueConstraintError`, `ErrCanvasNameAlreadyExists`
- `pkg/models/canvas_folder.go:540` — `mapCanvasFolderTitleUniqueConstraintError`
- `pkg/grpc/actions/canvases/common.go:88` — `errors.Is(...) → grpcerrors.AlreadyExists`

### Changes

**1. `pkg/models/user.go`**
- Add exported sentinel `ErrAPIKeyNameAlreadyExists = errors.New("api key name already exists")`.
- Add const for the constraint name `unique_api_key_in_organization`.
- Add `MapAPIKeyNameUniqueConstraintError(err error) error` that maps the
  matching `*pgconn.PgError` (`ConstraintName == unique_api_key_in_organization`)
  to the sentinel, else returns `err` unchanged.
- In `CreateAPIKey`, wrap the `tx.Create(user).Error` return through the mapper
  so callers see the sentinel regardless of the transaction wrapping.

**2. `pkg/grpc/actions/apikeys/create_api_key.go`**
- After `db.Transaction(...)` returns `err`, before the blanket `Internal`:
  ```go
  if errors.Is(err, models.ErrAPIKeyNameAlreadyExists) {
      return nil, grpcerrors.AlreadyExists(err, "an API key with this name already exists")
  }
  ```
- Because `Transaction` wraps the returned error, ensure the mapper runs
  *inside* the tx (in the model) so `errors.Is` still matches after wrapping.

**3. `pkg/grpc/actions/apikeys/update_api_key.go` (long-term robustness)**
- `UpdateAPIKey` also renames via `db.Save(user)` (line 72) and can hit the
  same constraint → same 500. Route its save error through the mapper +
  `AlreadyExists` too, so rename-to-duplicate is consistent with create.

### Tests

- `create_api_key_test.go` — add a case: create `ci-bot`, then create
  `" ci-bot "` (whitespace variant) → assert `codes.AlreadyExists` and a
  clear message; assert no second key exists.
- `update_api_key_test.go` — rename one key to another existing key's name →
  assert `codes.AlreadyExists`.

## Pros / Cons & Tradeoffs

**Pros**
- Reuses the DB unique index as the authority → no race window; correct even
  under concurrent creates.
- Matches existing conventions (`Canvas`, `CanvasFolder`) → low review risk,
  consistent API surface.
- Small, localized change; no migration, no proto change.

**Cons / Tradeoffs**
- Couples model code to a Postgres error type (`pgconn.PgError`). Already the
  accepted trade in this repo (`canvas.go`, `canvas_folder.go`).
- The mapper keys off the constraint name string; if the index is renamed the
  const must be updated in lockstep. Mitigated by keeping the name next to the
  sentinel and by a test that exercises the real DB path.
