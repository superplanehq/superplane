# Plan — Issue #4674: Custom roles not available in service account creation

## Problem
When creating a service account (surfaced in the product as an API Key under
Organization Settings > API Keys), the "Role" dropdown only offers the two
built-in roles: Viewer (`org_viewer`) and Admin (`org_admin`). Custom roles
created under Settings > Roles never appear, so a service account cannot be
granted a custom role.

The comment thread confirms the fix must cover both frontend and backend
(see PR #4700 referenced by @ramganesh-12).

## Root cause
The valid roles are hardcoded in two places:

1. Frontend — web_src/src/pages/organization/settings/ApiKeys.tsx (create modal)
   The role `<Select>` renders two literal `<SelectItem>`s:
     <SelectItem value="org_viewer">Viewer</SelectItem>
     <SelectItem value="org_admin">Admin</SelectItem>
   It never loads the org's roles.

2. Backend — pkg/grpc/actions/apikeys/create_api_key.go
   A hardcoded allow-list rejects anything but the two built-ins:
     validRoles := map[string]bool{RoleOrgAdmin: true, RoleOrgViewer: true}
     ... "invalid role for API key; must be org_admin or org_viewer"
   So even if the UI sent a custom role, the API would reject it.

Other creation flows already do this correctly. CreateGroupPage.tsx loads roles
via the `useOrganizationRoles` hook and renders them dynamically — we mirror
that proven pattern.

## Approach
Load roles dynamically on the frontend and validate against real roles on the
backend, while keeping the one intentional restriction: `org_owner` must not be
assignable to a service account (privilege-escalation guard, per the
service-accounts PRD, docs/prd/service-accounts.md).

### Backend (pkg/grpc/actions/apikeys/create_api_key.go)
Replace the hardcoded map with existence-based validation:
- Explicitly reject `models.RoleOrgOwner` with a clear message.
- Verify the role exists in the org using
  `authService.GetRoleDefinition(ctx, req.Role, models.DomainTypeOrganization, orgID)`;
  a lookup error means the role is invalid for this org.
This accepts any built-in (except owner) or custom org role, and is scoped to
the caller's organization so cross-org role names are rejected.

### Frontend (web_src/src/pages/organization/settings/ApiKeys.tsx)
- Load roles with `useOrganizationRoles(organizationId)` in the `APIKeys`
  component.
- Build a `sortedRoles` list (custom roles first, then built-ins), excluding
  `org_owner`, reusing the CreateGroupPage sort logic.
- Render `<SelectItem>`s from `sortedRoles` using
  `role.spec?.displayName || role.metadata?.name`.
- Keep `org_viewer` as the default when present; otherwise fall back to the
  first available role.

## Files to change
- pkg/grpc/actions/apikeys/create_api_key.go — role validation.
- pkg/grpc/actions/apikeys/create_api_key_test.go — cover custom role accepted,
  org_owner rejected, unknown role rejected.
- web_src/src/pages/organization/settings/ApiKeys.tsx — dynamic role dropdown.

## Flow
```mermaid
flowchart LR
  R[Roles page: create custom role] --> L[rolesListRoles / ListRoles]
  L --> UI[API Key create modal role dropdown]
  UI -->|CreateAPIKey role=custom| BE[create_api_key.go]
  BE -->|GetRoleDefinition ok and != org_owner| A[AssignRole]
```

## Pros / cons / tradeoffs
Pros
- Fixes both layers; custom roles work end to end.
- Reuses an existing, tested pattern (useOrganizationRoles + sortedRoles).
- Backend validates role existence per-org instead of a brittle allow-list;
  new built-in roles won't need code changes here.

Cons / tradeoffs
- One extra roles fetch on the API Keys page (cached by react-query, low cost).
- `org_owner` stays intentionally excluded; this is a deliberate security
  guard, not a limitation to remove.

## Testing
- Backend: `go test ./pkg/grpc/actions/apikeys/...`
- Frontend: build/typecheck; verify custom role appears and creation succeeds.
