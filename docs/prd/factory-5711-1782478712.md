# Cursor Launch Agent: Lifecycle Management and Parity with Claude runAgent

> Tracking issue: [#5711](https://github.com/superplanehq/superplane/issues/5711) â€” `enhancement`, `ai`

## Overview

The Cursor `launchAgent` component lets SuperPlane workflows launch and monitor a Cursor
Cloud Agent against a repository, optionally opening a pull request. It works, but it lacks
several lifecycle features that the Claude `runAgent` component (`pkg/integrations/claude/runagent/`)
already implements. This gap makes Cursor agents harder to use in reliable end-to-end workflows:
downstream nodes cannot see the full conversation, no resources are torn down on completion,
and the execution output omits the final message content.

This PRD specifies bringing the Cursor `launchAgent` component to parity with Claude `runAgent`
across three areas:

1. **Conversation history retrieval** â€” expose the full agent conversation, not just the last message.
2. **Session cleanup / teardown** â€” finalize the per-run agent session inline on completion, timeout,
   error, and cancellation (the node-scoped webhook and the branch/PR deliverables are out of scope).
3. **Execution metadata** â€” capture the agent summary and final message content in the execution
   output and metadata so they appear in the execution detail view without a separate
   `getLastMessage` call.

## Problem Statement

The Cursor `launchAgent` component currently has three concrete gaps relative to Claude `runAgent`:

### 1. Conversation history is limited to the last message

The only way to read what a Cursor agent produced is the separate `getLastMessage` action
(`pkg/integrations/cursor/get_last_message.go`). It calls the Cursor conversation endpoint via
`Client.GetAgentConversation` (`pkg/integrations/cursor/client.go`, `GET /v0/agents/{agentID}/conversation`),
which returns the **full** `ConversationResponse{ID, Messages[]}` â€” but the action discards everything
except `Messages[len-1]`:

```go
// get_last_message.go
if conversation.Messages != nil && len(conversation.Messages) > 0 {
    lastMessage := conversation.Messages[len(conversation.Messages)-1]
    output.Message = &lastMessage
}
```

Downstream nodes that need to reason about *what the agent did* (its intermediate steps, tool
calls, or reasoning) have no way to access that. By contrast, Claude `runAgent` assembles the full,
chronologically ordered message history in `Client.GetSessionMessages` and emits it as a `Messages[]`
array on every run.

### 2. No per-execution teardown on terminal paths

The component creates an ephemeral resource during a run â€” a Cursor Cloud Agent session â€” that is
never finalized when the run reaches a terminal state. `Client.CancelAgent`
(`POST /v0/agents/{agentID}/cancel`) exists, but it is only invoked from the manual `Cancel` path
(`LaunchAgent.Cancel`), never on terminal completion, timeout, or repeated error. A run that
completes, times out, or errors leaves no inline teardown step at all.

Note that `LaunchAgent.Cleanup(core.SetupContext)` is **not** the right lever here, and the gap is
not that it is a no-op. `Cleanup(SetupContext)` is the **node-deletion** hook (it fires when the node
is removed), not a per-execution terminal hook â€” and Claude's equivalent is *also* a no-op
(`pkg/integrations/claude/runagent/run_agent.go:338` returns `nil`). Claude does all its teardown
**inline** at each terminal path: the `Execute` fast-path (`run_agent.go:323-327`), poll completion
(`monitor.go:113-117`), timeout (`monitor.go:60-63`), repeated error (`monitor.go:81-82`), and
`Cancel` (`run_agent.go:153-155`). The Cursor gap is the absence of those inline terminal-path
teardown steps, *not* an unimplemented `Cleanup()`.

Other resources the run touches â€” the git branch (`cursor/agent-{exec_id[:8]}`), an optional PR, and
the status webhook â€” are **not** per-execution ephemeral resources and are out of scope for teardown:
the branch/PR are user-facing deliverables (see Non-Goals), and the webhook is node-scoped (registered
once in `Setup`; see Non-Goals and FR2).

Claude `runAgent`, in contrast, deletes the managed session, uploaded files, and temporary vault on
*every* terminal path. Sessions and side effects are not allowed to accumulate.

### 3. Execution output omits the final message content

The Cursor execution output payload is:

```go
// launch_agent_types.go
type LaunchAgentOutputPayload struct {
    Status     string `json:"status"`
    AgentID    string `json:"agentId"`
    PrURL      string `json:"prUrl,omitempty"`
    Summary    string `json:"summary,omitempty"`
    BranchName string `json:"branchName,omitempty"`
}
```

The execution metadata (`AgentMetadata`) already captures a `Summary` field (populated from the
webhook/poll payload), but **neither the metadata nor the output captures the agent's final message
content**. To see the actual final message, a workflow author must wire in a second `getLastMessage`
node. Claude `runAgent` instead surfaces `LastMessage` (and full `Messages[]`) directly on its
`OutputPayload`, so the final output is visible in the execution detail view with no extra node.

The combined effect: Cursor agents feel like a second-class citizen next to Claude agents, and
building robust multi-step workflows around them requires extra nodes and manual cleanup.

## Goals

1. Provide full conversation-history access for a Cursor agent run from within the `launchAgent`
   execution lifecycle, not only via a separate `getLastMessage` node.
2. Finalize the per-run Cursor agent session on all terminal paths (success, timeout, repeated error,
   cancellation) through a single idempotent helper with best-effort, non-fatal guarantees â€” calling
   `CancelAgent` only where the agent may still be running (the non-success paths), so a finished
   agent's deliverables are never disturbed.
3. Capture the agent summary and final message content in the execution output and metadata so they
   appear in the execution detail view without a follow-up `getLastMessage` call.
4. Keep the changes backward-compatible for existing workflows that already consume the current
   output fields and the standalone `getLastMessage` action.
5. Follow the patterns established in `pkg/integrations/claude/runagent/` so the two AI-agent
   components stay structurally consistent and maintainable.

## Non-Goals

- Changing the Cursor agent launch configuration surface (prompt, repository, branch, model,
  auto-create-PR, Cursor-bot options) beyond what is needed for these three features.
- Deleting or closing the git branch or PR that the Cursor agent created. These are intentional
  user-facing deliverables; teardown applies to the *ephemeral execution resource* â€” the per-run
  Cursor agent session â€” not to the agent's work product.
- Deregistering or tearing down the status webhook. The webhook is **node-scoped, not per-execution**:
  it is registered once in `Setup` via `ctx.Webhook.Setup()` (`launch_agent.go:108-110`) and routes
  every execution of the node via `FindExecutionByKV("agent_id", ...)`
  (`launch_agent_monitor.go:40`). Deregistering it on one execution's terminal path would break
  sibling and future runs of the same node, so the webhook is explicitly excluded from cleanup scope;
  its lifecycle is owned by the node (`Cleanup(SetupContext)` / platform webhook management).
- Adding cost/token usage metrics, per-step timing, or model-used auditing (useful, but out of scope
  for parity with the issue's three points).
- Reworking the polling/webhook monitoring architecture; we build on the existing
  `launch_agent_monitor.go` hook and webhook handler.
- Streaming conversation updates in real time during a run.

## Primary Users

- **Workflow Builders** who orchestrate Cursor Cloud Agents and need downstream nodes to act on the
  agent's full output and final message.
- **Platform / Ops** who care that Cursor agent sessions do not leak or accumulate across runs.
- **Support / Enablement** who debug failed or stuck agent executions and need the conversation and
  status visible in the execution detail view.

## User Stories

1. As a workflow builder, when a Cursor agent finishes, I want the final message available on the
   execution output so a downstream node (e.g. a notification or a conditional) can use it without
   adding a separate `getLastMessage` node.
2. As a workflow builder, I want access to the full conversation history of a Cursor agent run so
   downstream nodes can reason about the steps the agent took, not just its last line.
3. As a platform operator, I want each Cursor agent run to finalize its ephemeral resource (the
   per-run agent session) when it completes, times out, errors, or is cancelled, so sessions don't
   leak. (The node-scoped webhook is shared across runs and is not torn down per-execution.)
4. As a support engineer, I want the agent summary and final message visible in the execution detail
   view so I can diagnose a run at a glance.
5. As an existing user, I want my current workflows (consuming `status`, `agentId`, `prUrl`,
   `summary`, `branchName`, and the standalone `getLastMessage`) to keep working unchanged.

## Functional Requirements

### FR1 â€” Conversation history retrieval

- The component must be able to retrieve the **full** conversation for an agent run via
  `Client.GetAgentConversation` (`GET /v0/agents/{agentID}/conversation`), preserving message order
  and each message's `ID`, `Type`, and `Text`.
- The `launchAgent` terminal output must include the full conversation history (an ordered list of
  messages) in addition to the final message, mirroring Claude `runAgent`'s `Messages[]` field.
- Retrieval must tolerate eventual consistency: if the agent has reached a terminal status but the
  conversation is not yet fully written, the component must retry a bounded number of times before
  emitting. **This cannot mechanically mirror Claude's `GetSessionMessagesWithRetry`.** Claude's
  retry loops on `SessionMessages.Complete`, a flag set when a `session.status_idle` /
  `session.status_terminated` *event* appears in the event stream
  (`pkg/integrations/claude/runagent/client.go:267-272`). Cursor's `GET /v0/agents/{agentID}/conversation`
  returns only `ConversationResponse{ID, Messages[]ConversationMessage{ID,Type,Text}}`
  (`client.go:55-64`) â€” there is **no terminal/completion marker** to loop on. The component must
  therefore use a Cursor-specific completeness heuristic. The required v1 heuristic:
  - The agent's terminal status is already known from the webhook/poll completion path (it is the
    trigger for retrieval), so retrieval only runs once the agent is terminal.
  - "Conversation fully written" is detected by **stability**: read the conversation, wait
    `finalMessageDelay`, read again; if the message count (and the last message's `ID`) is unchanged
    across two consecutive reads, treat the history as complete. Retry up to `finalMessageReads`
    times; if still unstable, emit with the best history read so far and log that it may be partial.
  - An empty conversation on a terminal agent is retried (up to the bound) before giving up, since an
    empty read is the most likely eventual-consistency artifact.
  A retrieval failure (or a never-stabilizing conversation) must **not** block emitting the terminal
  status â€” the run still completes with whatever status/summary is known.
- The standalone `getLastMessage` action remains available and unchanged for backward compatibility.
- If the conversation history can be large, the implementation must handle pagination if/when the
  Cursor API exposes it; otherwise it must document the bound and log when results may be truncated.

### FR2 â€” Session cleanup / teardown

- **Scope of cleanup is the per-run Cursor agent session only.** The webhook is **not** a cleanup
  target â€” it is node-scoped (registered once in `Setup`, shared across executions) and tearing it
  down per-run would break sibling/future runs (see Non-Goals). The git branch / PR are deliverables
  and are never touched.
- **Cancellation semantics are split by whether the agent is terminal**, because `CancelAgent` on an
  already-finished agent risks aborting in-flight deliverable creation (e.g. an async `autoCreatePr`
  finalization where status is `FINISHED` while the PR is still being opened):
  - **Non-terminal terminal paths** â€” poll timeout (`attempt > LaunchAgentMaxPollAttempts`),
    repeated poll errors (`errors >= LaunchAgentMaxPollErrors`), and manual `Cancel` â€” the agent may
    still be running, so cleanup **must** call `Client.CancelAgent` to stop it (this is the existing
    `Cancel` behavior, now extended to timeout/error).
  - **Successful completion** (terminal status detected via webhook or poll) â€” the agent is already
    finished; teardown is a **no-op for the session** (do **not** call `CancelAgent`), so we never
    abort a PR that Cursor is still finalizing. v1 relies on Cursor's own lifecycle for finished
    agents. (If/when the Cursor API exposes an explicit terminate/delete that is safe on a finished
    agent â€” see Open Questions â€” it can be wired into this path; cancel is not that endpoint.)
- **A single shared `finalize` helper handles all terminal paths**, guarded by the real cross-path
  state, `ExecutionState.IsFinished()` â€” *not* by the persisted terminal status alone. This makes
  finalize **idempotent** so the webhook handler and the poll fallback (the two completion paths
  Cursor has, which Claude does not) cannot both run it. The helper, in order:
  1. Emits the terminal output (per FR3) â€” first, so a later step failing can never lose the result.
  2. Persists terminal metadata.
  3. Performs best-effort session teardown per the cancellation semantics above.
  Because emit precedes persist, the guard must be `IsFinished()` (which both paths and the framework
  honor), and teardown must be safe to skip if a prior path already finalized â€” preventing a
  double-`CancelAgent` across the webhookâ†”poll race.
- Cleanup must be **best-effort and non-fatal**: failures are logged via `ctx.Logger.Warnf` and do
  not fail the execution (matching Claude's `cleanupUploadedFiles` / `DeleteManagedSession` behavior).
- The agent ID needed for cleanup must be persisted in execution state/metadata early in `Execute`
  (it already is, via the `agent_id` KV and metadata) so finalize is possible on any error path,
  mirroring how Claude persists `uploaded_file_ids` and `managed_vault_id` immediately after creation.

### FR3 â€” Execution metadata and output

- The `launchAgent` terminal output payload must include:
  - The existing fields (`status`, `agentId`, `prUrl`, `summary`, `branchName`) â€” unchanged.
  - The agent's **final message content** (the last assistant message text).
  - The agent's **full conversation history** (per FR1).
- Execution metadata (`LaunchAgentExecutionMetadata` / `AgentMetadata`) must capture the agent
  summary (already present) and the final message content so both are visible in the execution detail
  view without a separate `getLastMessage` call.
- New output/metadata fields must be additive and optional (`omitempty`) so existing consumers and
  the example output snapshot remain valid; the `cursor.launchAgent.finished` payload type is
  preserved.
- Metadata and output population must go through the **single shared `finalize` helper** defined in
  FR2 (guarded by `ExecutionState.IsFinished()`), invoked identically from the **webhook handler**
  and the **poll fallback**. Both paths must produce the same final message, summary, and history
  regardless of which detected completion, and the shared guard prevents double-emit/double-cleanup
  across the webhookâ†”poll race. The existing `mergeWebhookPayloadIntoMetadata` / poll merge helpers
  feed metadata into `finalize`.

### FR4 â€” Consistency with Claude runAgent

- The implementation should reuse the structural patterns from `pkg/integrations/claude/runagent/`:
  full-history assembly with a Cursor-specific completeness heuristic (FR1), retry-on-incomplete,
  the shared idempotent `finalize` helper with emit-then-persist-then-teardown ordering (FR2), and
  best-effort non-fatal teardown.
- **Field shapes are decided here, not left to an Open Question.** Claude's output uses
  `messages []string` and `lastMessage string` (`runagent/types.go:58-59`); Cursor's native shape is
  objects (`ConversationMessage{ID,Type,Text}`). Reusing Claude's *names* while changing the *types*
  (string vs object) would make a downstream expression like `{{ $[...].data.lastMessage }}` resolve
  to a string for Claude but an object for Cursor â€” more confusing than distinct names. Therefore the
  Cursor output uses **distinct, type-honest field names** that preserve the richer object shape:
  - `lastMessage` â†’ the final message **object** (`ConversationMessage{id,type,text}`), keeping
    backward compatibility with the existing `getLastMessage` output shape (which already emits a
    `ConversationMessage`).
  - `messages` â†’ the **ordered array of `ConversationMessage` objects** (full history).
  This intentionally favors type-honesty and `getLastMessage` continuity over name-identity with
  Claude. The semantic parallel to Claude is documented in the field comments so users of both
  components understand the correspondence without assuming the types match.

## UX Requirements

- The execution detail view for a finished Cursor agent run shows the status, summary, final message,
  branch, and PR URL without the user adding any extra node.
- The full conversation history is available on the output for downstream wiring; presentation in the
  detail view should be readable (ordered messages, message type labels) and not overwhelm the
  default view.
- No new required configuration is introduced; existing `launchAgent` nodes continue to work as-is
  after upgrade.
- Cleanup is invisible to the user on the happy path; cleanup warnings appear only in execution logs.

## Implementation Scope (v1)

- **Conversation retrieval helper**: assemble the full ordered history from
  `Client.GetAgentConversation`, using the Cursor-specific stability heuristic and bounded
  retry-on-incomplete from FR1 (Cursor has no `Complete` flag to loop on).
- **Output and metadata extension**: add the `lastMessage` (object) and `messages` (object array)
  fields per FR4 to `LaunchAgentOutputPayload` and the execution metadata; populate them via the
  shared `finalize` helper from both the webhook and poll completion paths.
- **Shared finalize + cleanup**: implement the single idempotent `finalize` helper
  (emit â†’ persist â†’ session teardown) guarded by `ExecutionState.IsFinished()`; teardown calls
  `CancelAgent` only on the non-terminal paths (timeout, repeated error, manual cancel) and is a
  no-op on the success path. The webhook is **not** torn down. Persist the agent ID early in
  `Execute` (already done via the `agent_id` KV).
- **Tests**: add client-level tests (new cases in `client_test.go`) for history assembly and the
  stability/retry heuristic; extend `launch_agent_test.go` and `launch_agent_monitor_test.go` to
  cover metadata/output population from both completion paths, `finalize` idempotency under the
  webhookâ†”poll race, and that `CancelAgent` is called on non-terminal paths but **not** on the
  success path. (Note: there is no `get_last_message_test.go`; the cursor package's existing tests
  are `client_test.go`, `cursor_test.go`, `get_daily_usage_data_test.go`, `launch_agent_test.go`,
  `launch_agent_monitor_test.go`.)
- **Example output**: update `example_output_launch_agent.json` to include the new fields.

## Out of Scope (v1)

- Deleting/closing the agent's git branch or PR.
- Cost, token, or per-step timing metrics; model-used auditing.
- Real-time/streamed conversation updates during a run.
- Changes to the `getDailyUsageData` action or admin-key flows.
- Storing the prompt/instructions in metadata for replay (audit) purposes.

## Acceptance Criteria

1. A finished Cursor agent run emits an output that includes the final message content and the full
   conversation history, in addition to the existing `status`, `agentId`, `prUrl`, `summary`, and
   `branchName` fields.
2. The agent summary and final message are visible in the execution detail view without adding a
   `getLastMessage` node.
3. The conversation history is retrieved using the Cursor-specific stability heuristic with bounded
   retry when the agent is terminal but the conversation is not yet fully written (Cursor exposes no
   completion flag); a retrieval failure or never-stabilizing conversation still allows the run to
   complete with the known status/summary.
4. Teardown of the per-run agent session happens through the shared idempotent `finalize` helper on
   every terminal path; `CancelAgent` is called on the non-terminal paths (poll timeout, repeated
   poll errors, manual cancellation) but **not** on the success path (so a finished agent's in-flight
   PR is never aborted). The webhook is never torn down. Teardown failures are logged and never fail
   the execution, and the output is always emitted before persist/teardown.
5. Cleanup does not delete or close the agent's git branch or PR, and does not deregister the
   node-scoped webhook.
6. Existing workflows consuming current output fields and the standalone `getLastMessage` action
   continue to work unchanged; the `cursor.launchAgent.finished` payload type is unchanged and new
   fields are additive.
7. Output/metadata population is correct regardless of whether completion was detected via the
   webhook handler or the poll fallback, and `finalize` is idempotent across the two paths (no
   double-emit, no double-`CancelAgent`).
8. Unit tests cover history assembly + the stability/retry heuristic, metadata/output population from
   both completion paths, `finalize` idempotency under the webhookâ†”poll race, and that `CancelAgent`
   is invoked on non-terminal terminal paths but not on the success path.

## Success Metrics

- Reduction in workflows that need a separate `getLastMessage` node solely to read a Cursor agent's
  final output.
- Zero Cursor agent sessions left in a *running* state after a non-success terminal execution
  (timeout/error/cancel), measured via cleanup logs / Cursor-side session counts. (Successful runs
  are already finished, so they need no cancel.)
- Reduction in support cases citing "can't see what the Cursor agent did" or "stuck/leaked agent
  sessions."
- Parity checklist between `cursor/launchAgent` and `claude/runAgent` for the three issue areas is
  fully satisfied.

## Risks and Mitigations

- **Risk:** The Cursor API offers no explicit "delete completed session" endpoint, only `cancel`.
  **Mitigation:** Treat `cancel` as the teardown for non-terminal sessions only; for already-terminal
  sessions, rely on Cursor's own lifecycle and document the behavior (see Resolved Decisions).
- **Risk:** Conversation eventual consistency â€” terminal status arrives before the conversation is
  fully written. **Mitigation:** Bounded retry-on-incomplete (Claude pattern); never block emitting
  the terminal status on history availability.
- **Risk:** Large conversations bloat the output payload or detail view. **Mitigation:** Honor any
  API pagination; cap/log if necessary; keep the detail-view presentation collapsible.
- **Risk:** Cleanup runs before output is persisted and a failure loses the result. **Mitigation:**
  Strict emit-then-persist-then-teardown ordering inside the shared `finalize` helper.
- **Risk:** The webhookâ†”poll race (a hazard Claude lacks, since it has only a poll path) causes
  double-emit or double-`CancelAgent`. **Mitigation:** A single `finalize` helper guarded by
  `ExecutionState.IsFinished()`, idempotent and safe to invoke from either path; teardown is skipped
  if the run is already finalized. Tested explicitly.
- **Risk:** `CancelAgent` on a successfully-finished agent aborts in-flight PR creation, violating the
  deliverables Non-Goal. **Mitigation:** Success-path teardown is a no-op (no `CancelAgent`); cancel
  is reserved for the non-terminal paths where the agent may still be running.
- **Risk:** New output fields break the snapshot/example or strict consumers. **Mitigation:** Keep
  fields additive/optional and update `example_output_launch_agent.json`.

## Resolved Decisions

These were open during review and are now resolved in the FRs above; recorded here for traceability.

- **Cancel as the only lifecycle control (was OQ1).** v1 assumes `POST /v0/agents/{agentID}/cancel`
  is the only lifecycle control. Because cancel on a *finished* agent risks aborting in-flight
  deliverable creation, the success path does **not** call it (no-op session teardown, rely on
  Cursor's own lifecycle); cancel is used only on non-terminal paths. If a safe terminate/delete for
  finished agents is later confirmed, it can be wired into the success path without other changes.
- **Webhook is out of cleanup scope (was OQ3).** The webhook is node-scoped (registered once in
  `Setup`), so it is never deregistered per-execution; its lifecycle stays with the node. See FR2 and
  Non-Goals.
- **Output field shapes (was OQ4).** Distinct, type-honest names: `lastMessage` is a
  `ConversationMessage` object (matching the existing `getLastMessage` shape) and `messages` is an
  ordered array of `ConversationMessage` objects â€” *not* Claude's `string`/`[]string`. See FR4 for
  the rationale (avoiding same-name/different-type confusion across components).

## Open Questions

1. Does `GET /v0/agents/{agentID}/conversation` support pagination or a result cap for long
   conversations? If so, the history helper must page; if not, we must bound and log truncation.
2. Should the full conversation be embedded in the single `default` output channel, or surfaced via
   an additional output channel to keep the primary payload lean?
3. The issue lists "agent summary" as missing, but `AgentMetadata.Summary` is already captured today.
   Confirm whether the real gap is solely the **final message content**, or whether summary capture
   is also inconsistent on some completion paths and needs hardening.
