# Remove "Private Network Access" From the Self-Hosted Onboarding Flow

> Tracking issue: [#3946](https://github.com/superplanehq/superplane/issues/3946) — _"The first screen after setting up a self-hosted instance can't be a complex private network config"_ (label: `papercut`)

## Overview

When a user installs SuperPlane and signs up as the owner, the second screen of the
owner-setup wizard is a **"Private network access"** configuration step. It asks the brand-new
operator to make an SSRF / network-security decision — whether the instance may reach internal
services inside a VPC, private Kubernetes cluster, or other closed network — before they have
done anything else with the product.

This is the wrong decision to surface at the wrong time. The feedback on the tracking issue
(from Igor Sarcevic) is blunt:

> This should not be the first screen that a user sees after signing up in the self-hosted install:
> 1. Way too technical
> 2. Way too much words
> 3. I consider myself technically versed, and I would be totally confused on this page

This PRD proposes **removing the Private Network Access step from the onboarding wizard
entirely**, keeping the safe default (protection on), and relying on the already-existing
admin-settings page and environment variables as the home for this advanced setting. It also
proposes improving the discoverability of the setting for the small minority of operators who
genuinely need it, so that removing it from onboarding does not strand them.

## Background

### Where this lives today

The owner-setup wizard is a single React component, `OwnerSetup.tsx`, that renders one of five
steps based on local state (`web_src/src/pages/auth/OwnerSetup.tsx:26`):

| # | Step (`step` state) | Component | Purpose |
|---|---------------------|-----------|---------|
| 1 | `owner` | `OwnerStep.tsx` | Create the owner account (email, name, password) |
| 2 | `privateNetwork` | `PrivateNetworkStep.tsx` | **Toggle private network access (this PRD)** |
| 3 | `smtpPrompt` | `SmtpPromptStep.tsx` | Choose "Set up SMTP" or "Do this later" |
| 4 | `smtpConfig` | `SmtpConfigStep.tsx` | (Optional) Enter SMTP server details |
| 5 | `survey` | `PostHogSurveyForm` | (Optional) Post-setup PostHog survey |

The user reaches `/setup` only on a fresh install where owner setup has not yet completed; the
auth middleware redirects there when `OwnerSetupRequired()` is true. So this is, in practice, the
very first interaction an operator has with the product.

The flow transitions are linear and the private network step is **mandatory and unskippable** —
`handleOwnerNext` always advances to `privateNetwork`
(`web_src/src/pages/auth/OwnerSetup.tsx:188`), and `privateNetwork` always advances to
`smtpPrompt` (`:192`). There is no "skip" affordance on this step; the only controls are **Back**
and **Next** (`PrivateNetworkStep.tsx:58-65`).

### The exact copy on the screen today

From `web_src/src/pages/auth/ownerSetup/PrivateNetworkStep.tsx`:

- **Title:** "Private network access"
- **Subtitle:** "Decide whether this SuperPlane instance should be allowed to reach internal
  services before continuing to email setup."
- **Toggle label:** "Allow private network targets" (default: **Disabled**)
- **Help text:** "Enable this if SuperPlane needs to reach tools inside your VPC, private
  Kubernetes cluster, or another closed network. This reduces SSRF protection for private
  addresses."

This is four sentences of infrastructure/security jargon (VPC, Kubernetes, SSRF, private
addresses) presented before the user has any product context.

### What the setting actually controls

The toggle writes a single boolean, `allow_private_network_access`, into the installation
metadata:

- **Request field:** `allow_private_network_access`, sent to `POST /api/v1/setup-owner`
  (`web_src/src/pages/auth/OwnerSetup.tsx:132`).
- **Model:** `InstallationMetadata.AllowPrivateNetworkAccess bool`
  (`pkg/models/installation_metadata.go:17`), single-row table (`id = 1`), **default `false`**.
- **Enforcement:** `pkg/networkpolicy/http.go` — `ResolveHTTPPolicyForSetting()` reads the flag.
  When `false` (the default), it blocks a curated list of internal hostnames and private IP
  ranges (cloud metadata endpoints like `169.254.169.254`, `kubernetes.default*`, `localhost`,
  `10.0.0.0/8`, `192.168.0.0/16`, `fc00::/7`, etc.). When `true`, those block lists are cleared.

In other words, the **safe, secure default is `false`** (SSRF protection on). Doing nothing on
this screen is exactly the right choice for the overwhelming majority of installations.

### The setting already has two other homes

Critically, onboarding is **not** the only — or even the canonical — place to configure this:

1. **Admin settings page** (`web_src/src/pages/admin/InstallationSettings.tsx:264-334`):
   A "Network Policy" / "Private network access" section that an installation admin can change at
   any time via `PATCH /admin/api/installation/network-settings`
   (`pkg/public/admin.go:100-132`). It shows the effective blocked host/IP lists and whether an
   env override is in effect. Changing it invalidates the policy cache immediately.
2. **Environment variables** (`BLOCKED_HTTP_HOSTS`, `BLOCKED_PRIVATE_IP_RANGES`):
   These take precedence over the DB setting entirely. The README documents this as the
   infrastructure-level mechanism.

The precedence order is: **env vars → DB setting → hardcoded defaults**. The README already tells
operators they can _"enable private network access during owner setup **or later in
`/admin/settings`**"_ — so the post-onboarding path is already a documented, first-class option.

## Problem Statement

The Private Network Access step fails the onboarding job-to-be-done on three counts, all called
out in the issue:

1. **It is too technical for the audience and moment.** The first-run wizard is the worst place
   to ask a security-sensitive infrastructure question. The operator has zero product context and
   no way to know whether they need this. The terms (VPC, SSRF, private addresses, Kubernetes)
   assume deep platform-security fluency.
2. **It is too wordy.** Four sentences across the subtitle and help text, all explaining a setting
   that ~99% of installs should leave untouched.
3. **It is confusing even to technical users.** A self-described technically-versed reviewer was
   "totally confused" — because the screen demands a decision without giving the information
   needed to make it, and the consequences of getting it "wrong" (reduced SSRF protection) sound
   alarming.

Underlying root cause: **an advanced, rarely-needed, security-sensitive setting was placed on the
critical onboarding path as a mandatory step, despite already having a safe default and two other
configuration homes.** Onboarding should get the operator to a working product as fast as
possible; this step does the opposite.

## Goals

1. **Remove the Private Network Access step from the owner-setup wizard** so it is no longer the
   first (or any) screen new operators must clear.
2. **Preserve the secure default** (`allow_private_network_access = false`, SSRF protection on)
   for all new installations — removing the screen must not change behavior or weaken security.
3. **Keep the setting fully configurable** through its existing, more appropriate channels: the
   admin settings page and environment variables.
4. **Improve discoverability** so that the minority of operators who genuinely need private
   network access can find and enable it without friction — ideally guided by a helpful error at
   the moment they hit the SSRF block, rather than a pre-emptive wizard step.
5. **Shorten and simplify the onboarding flow** end to end.

## Non-Goals

- **Changing the network-policy enforcement engine.** `pkg/networkpolicy/http.go`, the block
  lists, env-var precedence, and the admin endpoint are out of scope and should keep working
  exactly as today.
- **Removing the admin settings control.** The `/admin/settings` "Private network access" section
  stays; it is the correct long-term home for this setting.
- **Changing the SMTP or survey steps.** This PRD only touches the private-network step and the
  wiring around it.
- **Redesigning the entire onboarding wizard.** A broader onboarding revamp may be worthwhile but
  is not in scope here; this is a targeted papercut fix.
- **Migrating existing installations.** Installs that already completed setup have their value
  persisted and are unaffected.

## Proposed Solution

### Recommendation: remove the step (Option A)

Remove the `privateNetwork` step from the wizard. New installs are created with
`allow_private_network_access = false` (the existing default). Operators who need private network
access enable it afterward in `/admin/settings` or via env vars — exactly as the README already
describes. Pair the removal with a **discoverability improvement** so nobody is stranded.

This directly resolves all three complaints: the technical screen is gone, the wordy copy is gone,
and there is no confusing decision on the critical path. The setting loses nothing in capability —
it already lives in admin settings and env vars.

### Options considered

**Option A — Remove the step from onboarding (recommended).**
The wizard goes `owner → smtpPrompt → [smtpConfig] → [survey]`. The setting defaults to off and is
managed post-onboarding.
- ✅ Fully resolves the issue; shortest flow; zero new copy to get wrong.
- ✅ No security regression — default is already the secure value.
- ⚠️ A day-one private-network user must discover the admin setting → mitigated by the
  discoverability work below (Goal 4).

**Option B — Keep the step but make it optional / progressive-disclosure.**
Collapse it behind an "Advanced" expander, default-collapsed, with a one-line summary and a
prominent "Skip" / "Use recommended defaults" action.
- ✅ Keeps the option visible in-flow for power users.
- ❌ Still adds a screen/affordance to the critical path; still has to carry the scary copy
  somewhere; partial fix to "too technical / too many words."

**Option C — Move it into a post-setup "first steps" checklist.**
Drop it from the required wizard and surface it as one card in a post-login getting-started
checklist.
- ✅ Removes it from the blocking path while keeping it discoverable.
- ❌ Requires building/owning a checklist surface that does not exist today; larger scope than a
  papercut warrants.

**Decision:** **Option A**, with the discoverability improvements from Option C's spirit applied
narrowly (error-message guidance + the existing admin page), because it is the smallest change
that fully fixes the reported problem without weakening security or losing functionality.

## Detailed Design

### Frontend changes (`web_src/src/pages/auth/`)

1. **Delete the step from the wizard state machine** in `OwnerSetup.tsx`:
   - Remove `"privateNetwork"` from the `step` union type (`:26`).
   - Remove the `{step === "privateNetwork" && <PrivateNetworkStep ... />}` render branch
     (`:255-264`).
   - Repoint `handleOwnerNext` to advance directly to `"smtpPrompt"` (currently `:188` sets
     `"privateNetwork"`).
   - Repoint `handleSMTPPromptBack` to go back to `"owner"` instead of `"privateNetwork"`
     (`:215`).
   - Remove the now-unused `handlePrivateNetworkNext` / `handlePrivateNetworkBack` handlers
     (`:191-199`).
2. **Decide the fate of the `allowPrivateNetworkAccess` state and request field.** Two viable
   approaches:
   - **Preferred:** Keep the `allow_private_network_access` field in the `POST /api/v1/setup-owner`
     body but hard-set it to `false`, so the API contract is unchanged and the value is explicit.
     Remove the `useState` toggle plumbing.
   - **Alternative:** Drop the field from the request entirely and let the backend default it to
     `false`. Requires confirming `setup_owner.go` treats an absent field as `false` (it should,
     given Go's zero value).
   The preferred approach is lower-risk because it does not change the wire contract.
3. **Delete `web_src/src/pages/auth/ownerSetup/PrivateNetworkStep.tsx`** and its imports.
4. **Update tests** that assert the private-network step appears in the flow or that reference
   `data-testid="owner-setup-private-network-switch"` (e.g. owner-setup E2E/unit specs). They
   should be updated to assert the step no longer renders and that `owner → smtpPrompt` is the
   transition.

### Backend changes (`pkg/public/setup_owner.go`)

- **No behavioral change required.** The endpoint continues to accept
  `allow_private_network_access`; when the frontend sends `false` (or omits it), the installation
  is created with SSRF protection on, identical to today's default.
- Optionally simplify by documenting that the field is now always `false` from the first-party
  UI, while leaving the field in place for API compatibility and for any scripted/automated setup
  that may rely on it.

### Discoverability improvements (Goal 4)

Removing the wizard step must not make the feature undiscoverable for the operators who need it.
Two low-cost, high-value improvements:

1. **Actionable SSRF-block error.** When an integration/component/trigger is blocked by the
   network policy (the `false` default), the error returned/logged should explicitly name the
   cause and the fix — e.g. _"This address is blocked by SuperPlane's private-network protection.
   An installation admin can allow private network access in Admin → Settings → Network Policy."_
   This turns the failure into self-service guidance at exactly the moment of need, which is far
   more effective than a pre-emptive wizard screen.
2. **Admin settings remains the clear home.** Confirm the `/admin/settings` "Private network
   access" section copy (`InstallationSettings.tsx:279-292`) is self-explanatory and that the
   section is easy to find. (It already shows the effective block lists and env-override notice.)

### Documentation

- Update `README.md`: the line _"enable private network access during owner setup or later in
  `/admin/settings`"_ should drop the "during owner setup" clause and point solely to
  `/admin/settings` and the env vars.

### Resulting onboarding flow

| Before | After |
|--------|-------|
| owner → **privateNetwork** → smtpPrompt → [smtpConfig] → [survey] | owner → smtpPrompt → [smtpConfig] → [survey] |

One mandatory step removed; the secure default is unchanged; the setting is unchanged and still
fully configurable post-setup.

## Edge Cases & Considerations

- **Day-one private-network requirement.** An operator whose first integration targets an internal
  host will hit the SSRF block immediately. Mitigation: the actionable error message (above) routes
  them to the admin setting in one step. This is strictly better than asking everyone the question
  up front.
- **Env-var-driven installs.** Operators who already set `BLOCKED_HTTP_HOSTS` /
  `BLOCKED_PRIVATE_IP_RANGES` are unaffected; env vars still take precedence over the DB flag.
- **Idempotency / re-running setup.** Removing the step must not change the "already initialized"
  (HTTP 409) behavior of `/api/v1/setup-owner`.
- **API/automation callers.** Any external script that posts to `/api/v1/setup-owner` with
  `allow_private_network_access: true` must keep working — hence the recommendation to retain the
  field in the contract rather than delete it.
- **Existing installations.** No migration needed; their persisted value is untouched.
- **Security review.** Because the default is already `false`, removing the screen cannot weaken a
  default install. The only way to weaken protection remains an explicit admin/env action.

## Rollout

- Single PR: frontend step removal + test updates + README copy, with the optional error-message
  improvement either in the same PR or a fast follow.
- No feature flag needed — this only removes a screen and preserves existing defaults and APIs.
- No data migration.

## Success Metrics

- **Qualitative:** The reported confusion is resolved — the first post-signup screen is no longer
  a network-security config. Re-confirm with the issue reporter.
- **Flow length:** Mandatory onboarding steps reduced from 2 (owner + private network) to 1
  (owner) before the SMTP decision.
- **Funnel (if instrumented):** Owner-setup completion rate holds or improves; no measurable
  increase in time-to-first-working-instance.
- **No regressions:** No increase in SSRF/security incidents; private-network adoption among the
  installs that need it remains achievable via admin settings (track via the admin
  network-settings PATCH usage, if available).

## Open Questions

1. **Retain vs. drop the `allow_private_network_access` request field?** Recommendation is to
   retain it (hard-set to `false`) for API stability — confirm no first-party caller depends on a
   user-provided value.
2. **Scope of the error-message improvement.** Is improving the SSRF-block error in-scope for this
   papercut PR, or a fast follow? Recommendation: include it if cheap, since it is what makes the
   removal safe for power users.
3. **Should the admin "Network Policy" section get a small copy polish** as part of becoming the
   sole in-product home for this setting, or is the current copy sufficient?
