# Cursor Launch Agent: Lifecycle Management & Parity with Claude `runAgent`

> Tracking issue: [superplanehq/superplane#5711](https://github.com/superplanehq/superplane/issues/5711)

## Overview

The Cursor `cursor.launchAgent` component launches a Cursor Cloud Agent, waits for it to
complete (via webhook + fallback polling), and emits a finished payload. It is the Cursor
counterpart to the Claude `claude.runAgent` component.

Over the course of building `claude.runAgent`, several lifecycle capabilities were added that the
Cursor component never received. This PRD defines the work to bring `cursor.launchAgent` (and the
companion `cursor.getLastMessage` component) to functional parity with `claude.runAgent` in three
areas: **conversation history retrieval**, **resource cleanup on completion**, and **richer
execution metadata**.

The goal is to make Cursor-based workflows as reliable, observable, and self-cleaning as Claude-based
workflows, so builders can mix and match agent providers without losing capabilities.

## Problem Statement

Workflows built on Cursor agents are harder to operate than the Claude equivalents:

1. **No access to the full conversation.** Downstream nodes can only retrieve the *single* last
   message (`cursor.getLastMessage`). To reason about *what the agent actually did* — the steps it
   took, intermediate reasoning, tool calls reflected in messages — there is no supported way to get
   the full transcript, even though the underlying Cursor API returns it.

2. **No teardown.** `claude.runAgent` deletes its managed session on terminal success and on cancel,
   and cleans up its uploaded files and temporary secrets vault on those paths (and on timeout/error,
   where it cleans files/vault but does *not* delete the session). `cursor.launchAgent` never deletes
   anything. Completed agents accumulate in the user's Cursor account with no automatic cleanup.

3. **Thin execution detail.** The execution detail view shows basic fields (agent ID, status, branch,
   PR URL). The agent *summary* is captured in metadata but not consistently surfaced, and the
   agent's *final message content* is not captured at all. To see what the agent concluded, an
   operator must add and run a separate `cursor.getLastMessage` node.

These gaps reduce reliability (orphaned resources), observability (can't see what happened without
extra nodes), and parity (Cursor is a second-class agent provider compared to Claude).

> Parity note: Claude emits a flattened `[]string` of assistant-message text. Cursor's native data model
> is the structured `ConversationMessage` (`{id,type,text}`) already used by `getLastMessage`. This PRD
> deliberately adopts the **Cursor-native `{id,type,text}` shape** for output and metadata (see A1 / FR
> resolution below) rather than copying Claude's `[]string` — internal consistency with `getLastMessage`
> wins over byte-for-byte parity with Claude. The "parity" goal is about *capabilities*, not identical
> output schemas.

Notable code facts that constrain the design:

- `client.GetAgentConversation(agentID)` **already returns the full `Messages` array**
  (`ConversationResponse.Messages []ConversationMessage`). `getLastMessage` deliberately discards all
  but `Messages[len-1]`. Full-history retrieval is therefore mostly a surfacing problem, not a new
  API integration.
- The Cursor client has **no delete method**. It exposes `CancelAgent` (`POST /v0/agents/{id}/cancel`)
  but not delete. The public Cursor API does provide `DELETE /v0/agents/{id}`
  ([docs](https://docs.cursor.com/en/background-agent/api/delete-agent)), returning `{ "id": "..." }`.
- `ConversationMessage` carries `{ id, type, text }` where `type` is `user_message` or
  `assistant_message`.
- `AgentMetadata` already has a `Summary` field that is populated by both
  `mergeWebhookPayloadIntoMetadata` and `mergeAgentResponseIntoMetadata`.
- **Cursor's `ConversationResponse` has no completeness signal.** It is exactly
  `{ id, messages []ConversationMessage }` (`client.go:61-64`). Unlike Claude's `SessionMessages`,
  which carries a `Complete bool` derived from `session.status_idle/terminated` events
  (`claude .../client.go:245,308`) and lets `poll` re-schedule while `!sm.Complete`
  (`monitor.go:99-102`), there is **nothing to retry against** in Cursor. The
  `GetSessionMessagesWithRetry` pattern therefore does **not** transfer (see A1 resolution).
- **Claude does not delete the session on the timeout / repeated-error paths.** `monitor.go` calls
  only `cleanupUploadedFilesFromHook` + `cleanupManagedVaultFromHook` there (lines 60-63, 81-82);
  `DeleteManagedSession` is invoked only on terminal-success (line 113) and cancel (line 153). Since
  Cursor has no uploaded-files/vault analogue, *faithful* parity on those paths means "no resource
  cleanup," not "delete the agent" (see B2 resolution).
- The Cursor completion paths differ structurally from Claude's. `HandleWebhook`
  (`launch_agent_monitor.go:14-94`) is a **single synchronous request that emits inline and cannot
  reschedule itself**; only `poll` can call `scheduleNextPoll`. For Cursor the webhook is the
  *primary* completion path and poll is the fallback (the reverse of Claude, which is poll-only).
  Any "fetch the transcript before emit" logic must therefore work without rescheduling on the
  webhook path (see A1 resolution).

## Goals

1. Provide a supported way to retrieve the **full conversation history** from a Cursor agent run.
2. **Clean up Cursor resources** when a run reaches a *terminal* state (Cursor reports the agent as
   `FINISHED`/`ERROR`/etc.) and on cancellation — mirroring `claude.runAgent`'s teardown discipline.
   Note: timeout and repeated-polling-error paths are **not** terminal on Cursor's side (the agent is
   still running), so they are handled separately (see B2) rather than via a bare delete.
3. Capture the agent **summary and final message content** in execution metadata so they are visible
   in the execution detail view without running a separate node.
4. Keep behavior **configurable and backward-compatible** so existing canvases keep working and so
   cleanup never silently destroys data a downstream node still needs.

## Non-Goals

- Adding webhook-based message streaming or live log tailing for Cursor agents.
- Reworking the launch/polling/webhook completion machinery (it works; we extend it).
- Building a UI for browsing full transcripts beyond what the execution detail view already renders.
- Parity for Claude-only concepts that have no Cursor equivalent (uploaded files, temporary vaults,
  secret injection at the egress layer). Cursor cleanup applies to Cursor-native resources only.
- Deleting branches, PRs, or repositories the agent created in the source provider.

## Primary Users

- **Workflow Builders** — compose Cursor agent steps and downstream logic; need transcript access and
  trustworthy completion data.
- **Operators / On-call** — inspect execution detail to understand what an agent did and why a run
  ended the way it did.
- **Account Owners** — care that automated runs don't leave orphaned agents/resources accumulating.

## User Stories

1. As a workflow builder, I want a downstream node to read the agent's *full* conversation so I can
   branch on intermediate steps, not just the final message.
2. As a workflow builder, I want the agent's summary and final message available on the
   `launchAgent` output and in the execution detail view without wiring a `getLastMessage` node.
3. As an account owner, I want completed Cursor agents and their transient resources cleaned up
   automatically so my account isn't littered with stale runs.
4. As an operator, when a run times out or fails, I want the failure reason to be visible in the
   output `status` (the agent itself is left in place since it is still running on Cursor's side —
   see B2).
5. As a workflow builder who *does* need the transcript after completion, I want control over
   whether cleanup deletes the agent (so I don't lose access to its conversation).

## Functional Requirements

### A. Conversation History Retrieval

Two complementary surfaces, mirroring how Claude exposes both a final message and the full transcript:

**A1. Full transcript on the `launchAgent` output (capability parity with Claude `OutputPayload.Messages`).**

- Extend `LaunchAgentOutputPayload` with a `messages` array of `{ id, type, text }`
  (`ConversationMessage` shape — the same structure `getLastMessage` already returns) plus a
  convenience `lastMessage` string (the `text` of the final `assistant_message`), alongside the
  existing `summary`. **Output-shape decision (resolves former Open Question #4): the structured
  `{id,type,text}` array is the contract**, not Claude's flattened `[]string`, for consistency with
  `getLastMessage`/`getConversation` and because Cursor's API returns this natively. `lastMessage` is
  derived, not a separate fetch.
- On terminal completion, fetch the conversation **once, best-effort** via
  `client.GetAgentConversation(agentID)` and populate the output. **The Claude
  `GetSessionMessagesWithRetry` pattern is explicitly NOT adopted**: Cursor's `ConversationResponse`
  has no `Complete`-style signal (see Current State), so a retry loop has nothing to test against and
  would only re-read the same buffer N times. A single fetch is the honest contract. (If a
  completeness heuristic is ever wanted, the only available one is "last message `type ==
  assistant_message` AND agent status is terminal" — still a guess; it is **deferred**, not assumed.)
- **Webhook vs. poll consistency (resolves reviewer concern on the two completion paths).** Both
  paths perform the *same* single best-effort fetch immediately before emit, so they produce the
  same output contract. We accept that A1/C1 are **best-effort on both paths**: because the webhook
  is synchronous and cannot reschedule, and because there is no completeness signal, a transcript
  that is not yet flushed at emit time yields fewer/empty `messages`. This is documented as expected
  behaviour, not a bug. The webhook path already issues a `GetAgentStatus` refresh when the summary
  is missing (`launch_agent_monitor.go:60-73`); the conversation fetch is added in the same place so
  the common (webhook) path is no worse than the poll path.
- Fetch failures must **degrade gracefully**: still emit the finished payload with status, agent ID,
  PR URL, branch, and summary; log a warning; leave `messages` empty and `lastMessage` blank. A
  transcript fetch must never fail the run.

**A2. A first-class full-conversation component (parity with the intent of `getLastMessage`).**

- Provide full-history retrieval as a supported downstream operation. Preferred approach: add a
  `cursor.getConversation` component that returns the entire `Messages` array; alternatively, extend
  `cursor.getLastMessage` with a mode/flag to return all messages.
- Output shape: agent ID plus an ordered `messages` array of `{ id, type, text }`.
- Reuse the existing `GetAgentConversation` client method; no new client API needed.
- Preserve `cursor.getLastMessage` behavior unchanged for backward compatibility.

> Design note: A1 and A2 overlap intentionally. A1 covers the common case (consume the transcript in
> the same run that launched the agent) without extra nodes; A2 covers retrieving history for an
> arbitrary agent ID, including across runs.

### B. Resource Cleanup

**B1. Delete the agent on terminal completion.**

- Add a `DeleteAgent(agentID string) error` method to the Cursor client calling
  `DELETE /v0/agents/{id}`.
- After a successful terminal emit — in **both** the webhook completion path
  (`HandleWebhook`) and the polling path (`poll`), and only when the agent's status is terminal —
  perform best-effort deletion (gated by `cleanupOnComplete`, see B5). Deletion failures are logged
  as warnings and must not change the emitted result, matching Claude's `DeleteManagedSession` error
  handling.

**B2. Timeout and repeated-polling-error paths (reconciled with B4 mitigation and actual Claude behaviour).**

- The poll path emits a `timeout` payload after `LaunchAgentMaxPollAttempts` and an `error` payload
  after `LaunchAgentMaxPollErrors`. On both, the agent is **non-terminal by definition** (it never
  reached a terminal status), so a bare `DELETE /v0/agents/{id}` would return 409 and clean up
  nothing — and it would directly contradict the "only delete after terminal status" mitigation (B4 /
  Risks).
- **Earlier drafts claimed this "mirrors `runagent/monitor.go`." That was wrong** and is corrected
  here: Claude's timeout (monitor.go:60-63) and repeated-error (monitor.go:81-82) paths call **only**
  `cleanupUploadedFilesFromHook` + `cleanupManagedVaultFromHook`; they do **not** call
  `DeleteManagedSession`. Cursor has no files/vault analogue, so *faithful* parity on these paths is
  to do **no resource cleanup at all**.
- **v1 decision:** the timeout and repeated-error paths perform **no deletion** (matching Claude).
  The agent is left in Cursor (it is still running and will reach its own terminal state there).
  Avoiding orphaned *stuck* agents would require a cancel-then-delete sequence (reusing B3); that is
  **deferred to a follow-up** and tracked in Open Questions, because it changes semantics (it would
  actively abort an agent the user's run merely stopped waiting on).

**B3. Clean up on cancel.**

- `Cancel` currently calls `CancelAgent` only. Extend it to also delete the agent after cancelling
  (best-effort), mirroring Claude's `Cancel` (interrupt → delete → cleanup files/vault).

**B4. Ordering and safety.**

- Cleanup happens **only after** a successful `Emit` of the finished payload (never before), so a
  failed emit can be retried without having destroyed state — exactly the Claude ordering.
- Cleanup is **best-effort and idempotent**: a 404 (already deleted) or other error is logged and
  swallowed; it must not block the workflow or cause re-emits.
- **Terminal status must be persisted (`Metadata.Set`) before deletion**, and the
  webhook-deletes-then-poll race must be handled explicitly. Scenario: the webhook completes, emits,
  and deletes the agent; a `poll` that was already scheduled then fires and calls `GetAgentStatus` on
  the now-deleted agent (`launch_agent_monitor.go:154`), which 404s and would increment `pollErrors`,
  potentially driving a spurious `error` emit. Today this is saved only by the early return at
  `launch_agent_monitor.go:122` (`isTerminalStatus(metadata.Agent.Status)`), which relies on the
  webhook having persisted terminal status via `Metadata.Set` (line 75) **before** the poll reads it.
  Once delete is added this ordering becomes load-bearing for *correctness*, so it is a requirement:
  1. The webhook persists terminal status to metadata **before** issuing `DeleteAgent`.
  2. A `GetAgentStatus` 404 (or other not-found) on an agent already known/recorded as terminal — or
     when the execution is already finished — **must not** be counted as a poll error; the poll must
     no-op (the existing line 122 early return is the intended guard and must remain effective).

**B5. Cleanup vs. transcript access (critical interaction).**

- Deleting the agent makes its conversation inaccessible (the `getLastMessage` docs already warn:
  "If the agent has been deleted, the conversation cannot be accessed"). Therefore:
  - If A1 is implemented, fetch the conversation for the output/metadata **before** deleting the
    agent.
  - Provide a config option on `launchAgent` to control deletion: `cleanupOnComplete`,
    **defaulting to `false`** (resolves former Open Question #1). When disabled (the default), the
    agent is retained so a downstream `getConversation`/`getLastMessage` node can still read it; when
    explicitly enabled, the agent is deleted on terminal success.
  - **Why `false` and not `true` (the AC3 vs AC9 conflict).** The *only* current way to read an
    agent's result is to wire a downstream `getLastMessage` node, which resolves the agent by ID at
    execution time via `GetAgentConversation`. Defaulting `cleanupOnComplete` to `true` would delete
    the agent on every existing canvas that has a downstream `getLastMessage`, turning those reads
    into 404s — precisely the breakage AC9 (existing canvases unchanged) forbids and the breakage
    `get_last_message.go`'s own doc warns about. Because A1/C1 degrade to empty on fetch failure, they
    cannot be *guaranteed* to replace that use case, so they do not justify a destructive default.
    Defaulting `false` lets AC3 and AC9 both hold: existing canvases are untouched, and operators who
    want auto-cleanup opt in. (Trade-off vs. Claude's always-clean behaviour is accepted and called
    out in UX Requirements.)
- Document that Cursor cleanup deletes the *agent record*; it does **not** delete branches, PRs, or
  environments the agent created in the source provider (Non-Goal). The issue's mention of "branches,
  environments" teardown is explicitly out of scope for v1 and called out in Open Questions.

### C. Execution Metadata

**C1. Capture the final message in metadata.**

- Add a single `LastMessage string` field to `AgentMetadata` (the `text` of the final
  `assistant_message`) and populate it from the fetched conversation on terminal completion (the same
  fetch used for A1), in both the webhook and poll completion paths.
- **Size cap (resolves former Open Question #5).** Metadata persists into every execution record, so
  the stored value is bounded: store **only the final message, not the full transcript**, and
  **truncate to 8 KB** (UTF-8 safe), appending a `…[truncated]` marker when cut. The full transcript
  lives on the output payload (A1) / is retrievable via `getConversation` (A2); it is **not** copied
  into metadata. This gives the execution-detail view (C3) the conclusion without bloating the row.

**C2. Ensure the summary is reliably surfaced.**

- `AgentMetadata.Summary` is already captured. Confirm it is rendered in the execution detail view;
  if the terminal webhook/poll lacks a summary, the existing `GetAgentStatus` refresh in
  `HandleWebhook` should backfill it. Apply the same refresh on the poll terminal path if needed.

**C3. Surface in the execution detail view.**

- Summary and final message should be visible in the execution detail view without running a
  separate `getLastMessage` node. Confirm the metadata fields are part of what the detail view
  renders (and add rendering if not).

## UX Requirements

- Existing `launchAgent` canvases continue to work with no reconfiguration; new output fields are
  additive.
- The execution detail view shows the agent summary and final message inline for completed runs.
- Any new cleanup configuration field has a sensible, clearly-described default and visible help text
  explaining the transcript-access tradeoff (B5).
- Failure states (timeout, repeated errors) remain distinguishable in the output `status`.

## Implementation Scope (v1)

- `client.go`: add `DeleteAgent(agentID)` (`DELETE /v0/agents/{id}`).
- `launch_agent_monitor.go`: invoke best-effort deletion (gated by `cleanupOnComplete`) after a
  successful emit on the **terminal** webhook and poll paths only; the timeout and repeated-error
  paths do **not** delete (B2); extend `Cancel` to delete after cancelling.
- `launch_agent_types.go`: extend `AgentMetadata` (final message) and `LaunchAgentOutputPayload`
  (messages + last message); add a `cleanupOnComplete` (or equivalent) spec field with a documented
  default.
- Single best-effort conversation fetch on terminal completion (NOT the
  `GetSessionMessagesWithRetry` retry pattern — Cursor has no completeness signal; see A1),
  populating output (A1) and the 8 KB-capped metadata field (C1), ordered before deletion (B5).
- Persist terminal status before deletion, and ensure a post-delete `GetAgentStatus` 404 is not
  counted as a poll error (B4).
- A2: add `cursor.getConversation` (preferred) or extend `getLastMessage` to return all messages;
  register the component in `cursor.go`.
- **Frontend (required for AC8):** `web_src/src/pages/app/mappers/cursor/launch_agent.ts`. The
  `Summary` row is *already* rendered by `getExecutionDetails` (line 74), so C2's "confirm rendered"
  is satisfied; the **final message is not** in `LaunchAgentMetadata` / `LaunchAgentPayloadData` (TS
  types, lines 16-27) and has no detail row. AC8 cannot be met by Go changes alone — this mapper must
  be edited to (a) add `lastMessage` to the payload/metadata TS types and (b) `addDetail` a "Final
  Message" row. (Earlier scope listed only Go files; this corrects that omission.)
- Update `example_output_launch_agent.json` and the `ExampleOutput()` to include new fields.
- Tests: cleanup invoked on each terminal path; cleanup ordered after emit; transcript fetched before
  delete; graceful degradation when fetch/delete fail; new component output; backward-compat of
  `getLastMessage`.

## Out of Scope (v1)

- Deleting branches/PRs/environments in the source provider.
- Streaming/live transcript updates.
- Retention policies or scheduled background cleanup of historical agents.
- Cross-provider abstraction of "agent lifecycle" beyond aligning these two components.

## Acceptance Criteria

1. A completed `launchAgent` run emits a payload that includes the agent summary and (when available)
   the final message and/or full conversation, in addition to status, agent ID, PR URL, and branch.
2. A supported downstream operation returns the **full** conversation history for an agent (new
   `getConversation` component or an all-messages mode on `getLastMessage`); `getLastMessage`'s
   existing single-message behavior is unchanged.
3. When `cleanupOnComplete` is enabled, a terminal-success run deletes the agent (best-effort) via
   `DELETE /v0/agents/{id}`, only after a successful emit and only after terminal status is persisted
   to metadata. `cleanupOnComplete` defaults to `false`, so by default no deletion occurs and AC9 holds.
4. On timeout and on repeated polling errors, the emitted status reflects `timeout` / `error` and the
   agent is **not** deleted (it is non-terminal; a bare delete would 409 — see B2). No regression in
   the emitted status.
5. On cancel, the agent is cancelled and then deleted (best-effort).
6. When deletion is disabled via configuration, the agent is retained and a downstream
   conversation-retrieval node can still read it.
7. Conversation/transcript fetch failures and deletion failures never fail the run; they are logged
   as warnings and the finished payload is still emitted.
8. The execution detail view shows the agent summary and final message for completed runs without a
   separate node. This requires the `launch_agent.ts` mapper change in Implementation Scope (summary
   row already exists; the final-message row and TS type field are added).
9. Existing canvases using `launchAgent` and `getLastMessage` continue to function unchanged.

## Success Metrics

- Reduction in orphaned/active Cursor agents per account after runs complete (target: agents deleted
  on the vast majority of terminal runs where cleanup is enabled).
- Reduction in workflows that add a `getLastMessage` node solely to inspect the result.
- No increase in run failure rate attributable to cleanup or transcript fetch (both are best-effort).
- Parity checklist (table in Current State) fully satisfied.

## Risks and Mitigations

- **Risk:** Deleting the agent destroys conversation access needed downstream.
  **Mitigation:** Fetch transcript before delete (B5); provide a config toggle to retain the agent;
  surface summary + final message in metadata so the common case needs no post-completion fetch.

- **Risk:** `DELETE /v0/agents/{id}` may reject deletion of a non-terminal/running agent (409).
  **Mitigation:** Only delete after terminal status (so the timeout/error paths do **not** delete —
  B2); treat 4xx/404 as best-effort no-ops with a warning.

- **Risk:** A scheduled poll fires after the webhook already deleted the agent and counts the
  resulting 404 as a poll error, driving a spurious `error` emit.
  **Mitigation:** Persist terminal status before delete and exempt not-found-on-terminal from the
  poll-error count (B4); the existing terminal-status early return (`launch_agent_monitor.go:122`) is
  the guard.

- **Risk:** Transcript not fully written at completion time, yielding partial/empty messages —
  especially on the synchronous webhook path.
  **Mitigation:** None available as a true fix (Cursor exposes no completeness signal), so A1/C1 are
  **best-effort by design** on both paths; a single fetch degrades to empty messages without failing
  the run. Documented as expected behaviour rather than mitigated away.

- **Risk:** Cleanup runs before emit and a transient emit failure leaves nothing to retry against.
  **Mitigation:** Strict ordering — cleanup only after a successful emit (B4), as in Claude.

- **Risk:** Scope creep into source-provider teardown (branches/PRs).
  **Mitigation:** Explicitly out of scope for v1; tracked as an open question.

## Resolved Decisions (previously open)

These were raised as open questions but are contract/correctness decisions that block implementation,
so they are decided here:

- **Cleanup default:** `cleanupOnComplete` defaults to **`false`** (preserve by default, opt in to
  clean) so AC3 and AC9 both hold. See B5.
- **Output message shape:** structured **`{id,type,text}` array** (`ConversationMessage`), not Claude's
  `[]string`, for consistency with `getLastMessage`/`getConversation`. `lastMessage` is a derived
  convenience string. See A1.
- **Metadata size:** store **only the final message, truncated to 8 KB**, never the full transcript.
  See C1.

## Open Questions

1. **A1 vs A2 priority:** Do we ship the transcript-on-output (A1) and the dedicated component (A2)
   together, or stage them?
2. **Source-provider teardown:** Is deleting agent-created branches/environments ever in scope, and if
   so, which provider integrations own that teardown?
3. **Orphan-avoidance on timeout/stuck agents (deferred from B2):** Should a future version
   cancel-then-delete an agent on the timeout / repeated-error path to avoid leaving a stuck agent
   running in Cursor? This changes semantics (it aborts an agent the run merely stopped waiting on),
   so it is out of v1 scope; do we want it at all, and behind what config?
