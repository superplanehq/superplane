# Self-Hosted First-Run Experience: Demote the "Private Network Access" Screen

> Tracking issue: [superplanehq/superplane#3946](https://github.com/superplanehq/superplane/issues/3946) — label: `papercut`

## Overview

When an administrator finishes signing up on a fresh self-hosted SuperPlane
install, the **second screen** of the owner-setup wizard is a dense,
security-jargon-heavy "Private network access" page about SSRF protection. This
PRD redefines the self-hosted first-run experience so that the first screens a
new admin sees are simple and welcoming, and the SSRF / private-network decision
is moved out of the critical onboarding path to a place where it belongs.

The goal is not to remove the control — SSRF protection and the ability to opt
into private-network targets remain essential — but to stop forcing a technical
infrastructure decision on a user who, at that moment, just wants to get into the
product.

## Problem Statement

The feedback that opened the issue (from Igor Sarcevic, internal review):

> This should not be the first screen that a user sees after signing up in the
> self-hosted install:
> 1. Way too technical
> 2. Way too much words
> 3. I consider myself technically versed, and I would be totally confused on this page

The screen in question reads:

- **Heading:** "Private network access"
- **Body:** "Decide whether this SuperPlane instance should be allowed to reach
  internal services before continuing to email setup."
- **Toggle:** "Allow private network targets" — *"Enable this if SuperPlane needs
  to reach tools inside your VPC, private Kubernetes cluster, or another closed
  network. This reduces SSRF protection for private addresses."*

### Why this is a problem

1. **Wrong altitude, wrong moment.** The very first impression of the product is
   a security-hardening decision (SSRF, VPC, link-local metadata endpoints) that
   most new users cannot evaluate yet — they have not connected anything, so they
   have no context for whether SuperPlane "needs to reach tools inside your VPC."
2. **High cognitive load.** The copy assumes the reader knows what SSRF is, what a
   private IP range is, and the security trade-off of relaxing it. Even
   technically versed reviewers report being confused.
3. **It blocks progress on an irreversible-feeling decision.** A "Next" button
   gates the rest of setup behind a choice the user is not equipped to make,
   creating hesitation at the worst possible time (first 60 seconds in product).
4. **The decision is premature.** The setting only matters once the user tries to
   connect an integration to a private target. At setup time there are zero
   integrations, so the default is almost always correct, and the rare user who
   needs it can flip it later.
5. **Redundant surface.** The exact same control already exists in the post-setup
   admin settings page, so the wizard step is not the only place to configure it —
   it is an avoidable detour.

### Current implementation (as-is)

Owner-setup wizard — `web_src/src/pages/auth/OwnerSetup.tsx` — step order:

| # | Step | Component | Required? |
|---|------|-----------|-----------|
| 1 | Admin account (email, name, password) | `ownerSetup/OwnerStep.tsx` | Yes |
| 2 | **Private network access** | `ownerSetup/PrivateNetworkStep.tsx` | **Forced (Next-gated)** |
| 3 | "Set up email delivery?" prompt | `ownerSetup/SmtpPromptStep.tsx` | Skippable |
| 4 | SMTP configuration (conditional) | `ownerSetup/SmtpConfigStep.tsx` | Skippable |
| 5 | PostHog survey (conditional) | `PostHogSurveyForm.tsx` | Optional |

Final submit: `POST /api/v1/setup-owner` (`pkg/public/setup_owner.go`) creates the
admin account, the default organization, and writes the
`allow_private_network_access` flag into the singleton `installation_metadata`
table.

Backend setting:

- **Model:** `pkg/models/installation_metadata.go` → `AllowPrivateNetworkAccess bool`
- **Column:** `installation_metadata.allow_private_network_access boolean DEFAULT false NOT NULL`
  (migration `20260331003955_add-installation-network-access-setting`)
- **Scope:** instance-wide singleton (`id = 1`)
- **Default:** `false` (SSRF protection ON — private ranges and metadata hosts blocked)
- **Enforcement:** `pkg/networkpolicy/http.go` (`ResolveHTTPPolicyForSetting`) +
  `pkg/registry/http.go` (URL/IP validation, DNS-rebinding guard) on every
  outbound `HTTPContext.Do()`.
- **Already configurable post-setup:** `web_src/src/pages/admin/InstallationSettings.tsx`
  → `GET/PATCH /admin/api/installation/network-settings`
  (`pkg/public/admin.go`). Updating it invalidates the policy cache so changes
  take effect within seconds — **no restart required.**

The key insight: **the secure default (`false`) is already the right answer for
essentially every brand-new instance, and the setting can be changed later, in
seconds, from a screen built for exactly that purpose.**

## Goals

1. Remove the "Private network access" screen from the forced, first-run owner-setup
   path so it is no longer the first/second thing a new admin sees.
2. Preserve the secure-by-default behavior: a fresh instance still blocks
   private-network / SSRF targets unless an admin explicitly opts in.
3. Ensure the private-network setting remains discoverable and changeable
   post-setup, with copy good enough that an admin can decide confidently when
   they actually need it.
4. Make the first-run experience short, welcoming, and free of decisions the user
   is not yet equipped to make.
5. Lose no security capability and require no schema or enforcement changes to the
   network policy itself.

## Non-Goals

- Changing the SSRF protection engine, the default blocklists, or the
  `networkpolicy` / `registry` enforcement logic.
- Changing the instance-wide default (`allow_private_network_access = false`).
- Removing the ability to opt into private-network targets — only relocating
  *when and where* that choice is presented.
- Reworking the SMTP / email-delivery steps (out of scope beyond reordering).
- Migrating existing installations' stored values (the column and its meaning are
  unchanged).
- Redesigning the full admin settings IA; only the network-access section copy is
  in scope.

## Primary Users

- **Self-hosted instance admins (first run):** technically capable operators who
  are installing SuperPlane for the first time and want to get into the product
  quickly.
- **Self-hosted instance admins (later):** the same operators when they eventually
  connect an integration that lives on a private network and need to relax SSRF
  protection deliberately.
- **Security-conscious operators:** admins who must understand and audit the SSRF
  posture of their instance.

## User Stories

1. As a new self-hosted admin, after I create my account I want to land in the
   product (or a short, friendly next step) — not a security configuration screen
   I can't yet evaluate.
2. As a new admin, I don't want to be forced to make an SSRF / private-network
   decision before I've connected anything.
3. As an admin who later needs SuperPlane to reach a tool in my VPC, I want to
   find and enable the private-network setting easily, with copy that tells me
   exactly what it does and what the trade-off is.
4. As a security-conscious operator, I want a fresh instance to be secure by
   default (private targets blocked) without anyone having to make a choice during
   setup.
5. As an admin, when I change the private-network setting I want it to take effect
   without restarting the instance.

## Proposed Solution

### Recommended approach: remove the forced step; rely on the secure default + post-setup admin page

1. **Remove `PrivateNetworkStep` from the owner-setup wizard flow.** New setup
   order becomes:
   - Step 1: Admin account
   - Step 2: "Set up email delivery?" prompt (skippable)
   - Step 3: SMTP configuration (conditional)
   - Step 4: PostHog survey (conditional)
2. **Keep the secure default.** `setup-owner` continues to persist
   `allow_private_network_access = false` (either by omitting the field and
   relying on the column default, or by sending `false` explicitly). No instance
   loses SSRF protection as a result of this change.
3. **Make the post-setup admin page the canonical home** for this decision
   (`InstallationSettings.tsx` → "Private network access" section already exists).
   Improve its copy (see Functional Requirements) so the decision is well
   supported where it actually belongs.
4. **Optionally, surface a contextual nudge** at the moment a user first tries to
   connect an integration to an address that the policy blocks — i.e., teach the
   setting when it becomes relevant, with a deep link to the admin setting. (See
   "Future / Optional Enhancements"; can ship after the core removal.)

This approach is the lowest-risk, highest-impact fix: it removes the papercut
entirely, changes no security behavior, and leans on a setting surface that
already exists and already hot-reloads.

### Alternatives considered

- **A. Soften the copy but keep the forced step.** Rewrite the screen to be
  friendlier and shorter. *Rejected as the primary fix* — it still forces a
  premature, out-of-context decision on the first screens; addresses complaints
  (1) and (2) but not (3) "wrong moment."
- **B. Make the step optional / collapsible ("Advanced") with a clear default.**
  Keep the step but make it skippable with one click and hide the detail behind a
  disclosure. Better than today, but still adds a screen to the critical path for
  a decision that is almost never needed at setup time.
- **C. Move it into the SMTP-style "do this later" pattern.** Present it as a
  prompt the user can defer. Equivalent in spirit to removal but keeps an extra
  screen; the admin page already provides the "later" path, so a dedicated prompt
  is redundant.

Recommendation: **ship the recommended approach (full removal from the forced
path).** If product wants to retain a setup-time touchpoint for discoverability,
fall back to alternative B (collapsed "Advanced" / one-click skip) rather than the
current forced screen.

## Functional Requirements

### FR1 — Owner-setup wizard

- The owner-setup wizard MUST NOT present the private-network access screen as a
  forced step in the default flow.
- After creating the admin account, the user MUST proceed directly to the email /
  SMTP prompt (or, if SMTP is also deferred, to the product/dashboard).
- "Back"/"Next" navigation and step indicators MUST be updated to reflect the new
  step count; no dead step or empty placeholder may remain.
- The wizard MUST continue to function end-to-end if the private-network choice is
  never made.

### FR2 — Default security posture preserved

- On completion of `POST /api/v1/setup-owner`, `allow_private_network_access` MUST
  be `false` for a new instance (SSRF protection ON).
- The behavior MUST be identical whether the field is omitted (column default) or
  explicitly sent as `false`.
- No change to `pkg/networkpolicy` defaults (`DefaultBlockedHTTPHosts`,
  `DefaultBlockedPrivateIPRanges`) or enforcement.

### FR3 — Post-setup configurability (canonical home)

- The admin installation settings page MUST retain a clearly labeled
  "Private network access" control bound to
  `GET/PATCH /admin/api/installation/network-settings`.
- Changing the setting MUST take effect without an instance restart (policy cache
  invalidation already implemented — verify it still fires).
- The page MUST surface the effective state, including when the value is overridden
  by the `BLOCKED_HTTP_HOSTS` / `BLOCKED_PRIVATE_IP_RANGES` environment variables
  (these already exist in the response payload: `*_overridden`, `effective_*`).

### FR4 — Improved copy on the admin setting

The admin "Private network access" section SHOULD be rewritten to be plain and
actionable. Suggested direction:

- **Label:** "Allow access to private networks"
- **Short helper:** "By default, SuperPlane can only reach public internet
  addresses. Turn this on only if you need SuperPlane to connect to tools on a
  private network — for example a service inside your own VPC or Kubernetes
  cluster."
- **Trade-off note (secondary text / tooltip):** "Allowing private addresses
  reduces protection against SSRF attacks. Leave this off unless you have a
  specific integration that requires it."
- Default/secure state SHOULD be visually obvious ("Recommended: off").

### FR5 — Backend request handling

- `SetupOwnerRequest.AllowPrivateNetworkAccess` MAY be retained for API
  compatibility but MUST default to `false` when absent. If the field is fully
  removed from the UI, the backend MUST still default safely.
- No database migration is required (column and default are unchanged).

### FR6 — Tests

- Update `test/e2e/owner_setup_test.go` to reflect the removed step; the existing
  assertions that drive `owner-setup-private-network-switch` MUST be removed or
  relocated.
- Add/adjust coverage proving a fresh instance ends setup with
  `allow_private_network_access = false`.
- Retain/extend `web_src/src/pages/admin/InstallationSettings.spec.tsx` coverage
  for toggling the setting and for the env-override display.

## UX / Copy Notes

- First-run should feel like a welcome, not a security checklist. After account
  creation, prefer momentum (email prompt → product) over configuration screens.
- Where the setting does appear (admin page, and any future contextual nudge),
  lead with the user's intent ("connect to a tool on my private network") rather
  than the threat model ("SSRF"). Mention SSRF as the *why* in secondary text.
- Use product naming "SuperPlane" (not "Superplane") in all copy.

## Future / Optional Enhancements

- **Contextual just-in-time prompt:** when an integration/component first attempts
  an outbound request that the policy blocks because it targets a private address,
  show an actionable error that explains the block and deep-links an admin to the
  "Private network access" setting. This teaches the setting precisely when it
  becomes relevant and is the ideal long-term replacement for a setup-time screen.
- **Per-integration / per-target allowlisting** (instead of an instance-wide
  switch) so admins can relax SSRF for a single trusted endpoint rather than
  globally. Larger effort; tracked separately.

## Success Metrics

- The private-network screen no longer appears in the default first-run flow
  (qualitative: re-review by the original reporter passes).
- No regression in default security posture: new instances measured at
  `allow_private_network_access = false`.
- Time-to-first-meaningful-screen after account creation is reduced (one fewer
  forced step).
- No increase in support requests of the form "SuperPlane can't reach my internal
  service," because the admin setting + (optional) contextual prompt make the fix
  discoverable when needed.

## Risks & Mitigations

- **Risk:** A user who genuinely needs private access at setup time no longer sees
  the toggle there. **Mitigation:** the admin page already exists, hot-reloads,
  and (optionally) a contextual prompt surfaces it exactly when an integration is
  blocked.
- **Risk:** API consumers depend on `setup-owner` accepting the field.
  **Mitigation:** keep the field optional and backward-compatible; default to
  `false` when absent.
- **Risk:** Discoverability of the setting drops. **Mitigation:** improved admin
  copy (FR4) and the optional just-in-time prompt.

## Open Questions

1. Do we fully remove `PrivateNetworkStep` from the wizard, or keep it behind a
   one-click "Advanced (optional)" disclosure for discoverability?
2. Should the `allow_private_network_access` field be removed from
   `SetupOwnerRequest` entirely, or retained (optional, defaulting `false`) for
   API compatibility?
3. Is the just-in-time "blocked outbound request" prompt in scope for this
   papercut, or a fast follow?
4. Any analytics we want to capture on how often admins enable the setting
   post-setup, to validate that the setup-time screen was indeed unnecessary?
