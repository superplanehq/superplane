# Running on Windows with WSL2

The development environment is built around `make dev.setup` and `make dev.start` (see [CONTRIBUTING.md](../../CONTRIBUTING.md)). On Windows, the supported way to run those targets is through **WSL2 + Docker Desktop**. This page documents the prerequisites, the recommended repository layout, and the issues most commonly reported by Windows contributors.

## Prerequisites

- Windows 10/11 with **WSL2** enabled and a Linux distribution installed (Ubuntu 22.04 or 24.04 is recommended).
- **Docker Desktop** with the *Use the WSL 2 based engine* and *Enable integration with my default WSL distro* options enabled (Settings → Resources → WSL integration).
- `make`, `git`, and Docker Compose available inside the WSL distribution.

Verify the toolchain from the WSL shell:

```sh
make --version
git --version
docker compose version
```

## Recommended: clone inside the WSL filesystem

Cloning the repository **inside the Linux filesystem** (for example under `~/`) is strongly recommended over cloning under `/mnt/<drive>/`. The WSL `/mnt/` paths are reached through the 9p protocol, which is significantly slower for the I/O patterns produced by Go module downloads, Vite dependency optimization, and frequent file watching.

```sh
# Run from inside WSL
git clone https://github.com/<your-username>/superplane.git ~/superplane
cd ~/superplane
make dev.setup
make dev.start
```

The application is then reachable at <http://localhost:8000>. Files edited from VS Code with the *WSL* extension or from any editor that targets the WSL distribution will trigger the usual hot-reload provided by `air` (Go) and Vite (frontend).

## Alternative: cloning under `/mnt/<drive>/`

The repository can also live on the Windows filesystem (for example `D:\repos\superplane`) and be reached from WSL through `/mnt/d/repos/superplane`. This setup works, but I/O on `/mnt/<drive>/` is served by the 9p protocol and is noticeably slower than native ext4. The first `make dev.setup` and the first cold-start of Vite take longer, and each container operation that touches the source tree is more expensive than on a native Linux filesystem.

The development Docker Compose file already accounts for the most common 9p limitations: build caches that require atomic file operations (`go-pkg-cache`, `uv-cache`, and `vite-cache`) live on Docker-managed volumes rather than on the bind-mounted source tree. No additional configuration is required.

## Troubleshooting

### `make dev.setup` fails on `docker pull` with "error getting credentials"

A Docker config generated by Docker Desktop on Windows may set:

```json
{ "credsStore": "desktop.exe" }
```

If the helper is not reachable from the WSL distribution, image pulls fail with:

```
docker: error getting credentials - err: exit status 1, out: ``
```

Removing the `credsStore` entry from `~/.docker/config.json` inside WSL allows Docker to fall back to the inline credentials (or to anonymous pulls for public images):

```sh
cp ~/.docker/config.json ~/.docker/config.json.bak
python3 -c '
import json, pathlib
p = pathlib.Path.home() / ".docker/config.json"
cfg = json.loads(p.read_text())
cfg.pop("credsStore", None)
p.write_text(json.dumps(cfg, indent=2))
'
```

`make dev.setup` can then be re-run; previously completed steps are skipped.

### Sub-`make` invocations report `getcwd: No such file or directory`

When the source tree is mounted from `/mnt/<drive>/`, long-running `make` recursions sometimes lose their working directory and exit with:

```
make: getcwd: No such file or directory
```

This is transient. Re-running the failed target from a fresh WSL shell is sufficient; `make dev.setup` is idempotent and resumes from the failing step.

### Inspecting application logs

When the UI fails to load or the API returns unexpected responses, the application logs are the first place to look:

```sh
make dev.logs.app
```

For the full set of services (`app`, `agent`, `db`, `rabbitmq`):

```sh
make dev.logs
```
